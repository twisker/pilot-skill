# PILOT 冲突检测规则（确定性规则清单）

> 本文件是冲突规则的**规范文本**（唯一来源）。规则的代码实现在 `tools/check.ts`（`run --trip <id>`）。
> C-01 ~ C-07 自 W5 SKILL.md 原样迁移（v3.0 时代文本，未改一字）；R-08 / R-09 为 v2（v4.0）新增。

---

## W5 原样迁移：C-01 ~ C-07

每次行程编辑后，按以下规则清单逐条检查。规则是确定性的，不使用 LLM 判断。

### C-01: 单日驾车时间超限（类别：距离）
检查每天的 transport_notes 中提到的驾车时间，如果单日累计超过 4 小时：
```
⚠️ 冲突 C-01: 第{N}天驾车时间约{X}小时，超过安全建议的4小时。
建议：拆分为两天，或删减沿途景点减少绕路。
```

### C-02: 时间反转（类别：时间）
检查每天的 POI 序列，如果后一个 POI 的 time_start 早于前一个 POI 的 time_end：
```
⚠️ 冲突 C-02: 第{N}天「{POI-A}」结束时间 {end} 晚于「{POI-B}」开始时间 {start}。
建议：调整时间顺序，或删除其中一个景点。
```

### C-03: 单日景点过多（类别：节奏）
检查每天的 POI 数量，如果超过 6 个（不含交通和住宿）：
```
⚠️ 冲突 C-03: 第{N}天有 {X} 个景点，行程过于密集。
建议：删减到 4-5 个景点，或拆分到相邻天。
```

### C-04: 住宿缺失（类别：后勤）
检查非首日和非末日的每一天，如果没有 category="住宿" 的 POI：
```
⚠️ 冲突 C-04: 第{N}天没有安排住宿。
建议：在当天最后添加住宿点。
```

### C-05: 总天数不匹配（类别：一致性）
检查 days 数组长度是否与用户声明的天数一致：
```
⚠️ 冲突 C-05: 用户要求 {X} 天，但当前行程有 {Y} 天。
建议：{增加/删减}天数以匹配需求。
```

### C-06: 首日出发时间过早（类别：可行性）
如果第一天第一个景点的 time_start 早于 09:00（考虑到达目的地通常在上午）：
```
⚠️ 冲突 C-06: 第1天首个景点安排在 {time}，到达目的地后可能来不及。
建议：调整到 10:00 之后，或把该景点移到第二天。
```

### C-07: 同行人群适配（类别：安全）
如果 travelers_context.with_elderly=true 且行程包含标记为高强度的活动：
```
⚠️ 冲突 C-07: 行程包含「{POI}」，可能不适合同行老人。
建议：替换为强度较低的替代景点。
```

## v4 字段对照说明（实现 check.ts 时使用，不改变规则语义）

上述 C-01 ~ C-07 文本写于 W5（v3.0 种子库时代），字段名对应旧 session schema。
v4.0 实现基于 `shared/schema/itinerary.schema.json` 与 `shared/schema/intake.schema.json`，对照如下：

| W5 文本中的字段 | v4 对应字段 |
|----------------|------------|
| transport_notes | `days[].items[]` 中 `kind="transit"` 条目的 `note`（驾车时长从 note 文本解析） |
| POI 的 time_start / time_end | `days[].items[].time`（v4 每条目单时间点；「时间反转」= 同日后一条目 time 早于前一条目 time） |
| POI 数量（不含交通和住宿） | `days[].items[]` 中 `kind` 为 `sight`/`meal`/`other` 的条目数（排除 `transit`/`hotel`） |
| category="住宿" 的 POI | `days[].items[]` 中 `kind="hotel"` 的条目 |
| 用户声明的天数 | `intake.dates` 的 start~end 天数（含首尾） |
| 第一天第一个景点的 time | `days[0].items[]` 中首个 `kind="sight"` 条目的 `time`（C-06 仅检查景点，transit/hotel 不算） |
| travelers_context.with_elderly | `intake.party.seniors > 0` |
| 高强度活动 | 条目 `note` 含高强度关键词（徒步/攀登/骑行/高海拔等，实现时冻结关键词表） |

## v2 新增：R-08 / R-09

### R-08: 单日花费超预算基线（类别：预算）
计算每天 `items[].cost_cny` 合计（null 不计入），若超过 `intake.budget_cny / 行程天数 × 3`：
```
⚠️ 冲突 R-08: 第{N}天预计花费 ¥{X}，超过日均预算基线（¥{budget_cny/天数}）的 3 倍。
建议：更换高消费条目，或与用户确认预算是否需要上调。
```
前提：`intake.budget_cny` 非 null；为 null 时本规则跳过。

### R-09: 相邻坐标距离异常（类别：坐标/幻觉拦截）

**2026-07-05 主控裁定**：原规则只查同日内相邻 geo 条目，对话编辑造成的跨天坐标
跳跃（如把某天的目的地换成一个遥远的地点）检测不到。规则扩为两段：

**同日内**：对每天内相邻两个 `geo` 非 null 的条目，计算直线（大圆）距离，若 > 300km：
```
⚠️ 冲突 R-09: 第{N}天「{A}」→「{B}」直线距离约 {X}km（>300km），坐标疑似错误。
建议：核对两点经纬度（geo 坐标必须来自 WebFetch 查证，禁止凭记忆填写）；确认无误则说明行程本身跨度过大。
```

**跨天**：对每一对相邻天（第 N 天 → 第 N+1 天），取第 N 天最后一个 `geo` 非 null
的条目与第 N+1 天第一个 `geo` 非 null 的条目，计算直线距离，若 > 600km（跨天含
正常长途驾驶转场日，阈值比同日内更宽松）：
```
⚠️ 冲突 R-09: 第{N}天「{A}」→第{N+1}天「{B}」跨天直线距离约 {X}km（>600km），坐标疑似错误。
建议：核对两点经纬度（geo 坐标必须来自 WebFetch 查证，禁止凭记忆填写）；确认无误则说明行程本身跨度过大（长途转场日）。
```
任一天没有 `geo` 非 null 的条目时，该对相邻天跳过跨天检测。

本规则的主要目的：拦截幻觉坐标——单日内两点直线 300km、跨天两点直线 600km 以上，
在自驾行程中几乎必然是查证失误或行程错误。两个阈值实现时均为具名常量
（`R09_SAME_DAY_THRESHOLD_KM` / `R09_CROSS_DAY_THRESHOLD_KM`）。

## 输出契约（check.ts run）

`tools/check.ts run --trip <id>` 输出违规数组：

```json
[{ "day": 3, "rule": "C-01", "severity": "warn", "detail": "第3天驾车时间约5.5小时，超过安全建议的4小时。" }]
```

- 所有规则均为 warn 级（不阻塞流程）；由 Skill 呈现给用户，用户可确认忽略。
- 规则实现（Task 10）以本文件为规范文本；如实现与文本冲突，以本文件为准并回改实现。
